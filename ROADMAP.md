## 1. Architecture Summary

```
┌──────────────────────────────────────────────────────┐
│  Next.js 15 (Vercel) — frontend only                 │
│  ├─ App Router (pages, layout)                       │
│  ├─ react-konva canvas (board view)                  │
│  ├─ SpacetimeDB React hooks (useTable, useReducer)   │
│  └─ /api/tmdb/search  ← one Route Handler (proxy)   │
└───────────────────┬──────────────────────────────────┘
                    │  WebSocket (SpacetimeDB SDK)
┌───────────────────▼──────────────────────────────────┐
│  SpacetimeDB Module (TypeScript) — Maincloud          │
│  ├─ Tables: Account, Board, Participant,              │
│  │          MediaItem, WatchEntry, WatchAggregate     │
│  ├─ Reducers: board CRUD, join, watch, bulk           │
│  └─ Procedures: import_tmdb_title (TMDB HTTP call)   │
└──────────────────────────────────────────────────────┘
       │
       └─ SpacetimeAuth (Google OAuth → OIDC token)
```

**Key architectural properties:**
- Next.js is a pure frontend — no DB access from Server Components or Route Handlers (except the TMDB proxy)
- All board data arrives via WebSocket subscriptions (`useTable`)
- Real-time updates are built-in from v1 — no polling, no SSE setup required
- Board pages are client-rendered (no SSR for dynamic board data)

## 3. Workstreams

| # | Workstream | Primary area | Phase 1 deliverable |
|---|---|---|---|
| W1 | Module scaffold | `module/src/lib.ts`, tables | All tables defined; module deploys to local spacetime server |
| W2 | Core reducers | `module/src/reducers/` | All reducers implemented and tested locally |
| W3 | TMDB procedure | `module/src/procedures/tmdb.ts` | `import_tmdb_title` fetches and inserts MediaItem tree |
| W4 | Next.js scaffold | `app/`, `providers.tsx` | SpacetimeDB connection live in browser; auth flow working |
| W5 | Board management UI | `app/board/`, board reducers | Create/view/delete boards; invite link generation |
| W6 | TMDB search UI | `components/board/AddMediaSearch`, `/api/tmdb/search` | Search + trigger import; MediaItems appear in subscription |
| W7 | Canvas rendering | `components/canvas/`, `lib/canvas-layout.ts` | Nodes rendered on react-konva canvas; pan + pinch-zoom |
| W8 | Zoom collapse | `ZoomController`, node components | 3-level collapse animated on zoom |
| W9 | Watch tracking | `set_watch` reducer, `useWatchState` | Mark watched per title/episode; state pushes to all clients |
| W10 | Watch overlay | `WatchIndicator` | Per-participant chips on each node; aggregate on collapsed |
| W11 | Participant join | `join_board` reducer, `app/board/[id]/join/` | Guest joins via invite link; anonymous identity persisted |
| W12 | Sharing modes | `update_board` reducer, settings page | PRIVATE / PUBLIC toggle; public view-only route |

W1–W2 are prerequisites for W3–W6. W4 is prerequisite for W5–W12. W6 is prerequisite for W7. W7 is prerequisite for W8, W10. W9 depends on W7 + W11.

## 4. Phase 1 Milestones

### M0 — Environment setup (days 1–2)

- [ ] Install SpacetimeDB CLI: `curl -sSf https://install.spacetimedb.com | sh`
- [ ] `spacetime init --lang typescript module` — scaffold the module
- [ ] `npx create-next-app` with TypeScript + Tailwind + App Router
- [ ] Install frontend deps: `spacetimedb`, `react-konva`, `konva`
- [ ] `.env.local.example` with all required variables documented
- [ ] Vercel project created; connected to repo
- [ ] SpacetimeDB Maincloud project created; SpacetimeAuth configured with Google provider
- [ ] `spacetime start` runs locally; `spacetime publish` deploys to Maincloud

### M1 — Module: tables + reducers (days 2–5)

- [ ] All 6 tables defined in `module/src/tables/`
- [ ] All reducers implemented in `module/src/reducers/`
- [ ] Module deploys locally; `spacetime generate` produces `module_bindings/`
- [ ] Manual reducer tests via `spacetime call` CLI

### M2 — Module: TMDB procedure (days 5–7)

- [ ] `import_tmdb_title(board_id, tmdb_id, tmdb_type)` procedure implemented
- [ ] Fetches movie or TV show (with seasons + episodes) from TMDB API
- [ ] Inserts `MediaItem` tree with correct `parent_id` hierarchy
- [ ] `chrono_order` initialised from `air_date`
- [ ] Procedure tested locally against TMDB sandbox

### M3 — Next.js: connection + auth (days 7–9)

- [ ] `SpacetimeDBProvider` configured in `app/providers.tsx`
- [ ] `lib/spacetimedb.ts`: `DbConnection` singleton with connect/disconnect lifecycle
- [ ] SpacetimeAuth Google login flow at `/login` — redirects to SpacetimeAuth, returns OIDC token, stores in `localStorage`
- [ ] On connect with token: `register_owner` reducer called if `Account` row doesn't exist
- [ ] `useParticipant` hook: detects if current identity is owner or participant

### M4 — Board management (days 9–12)

- [ ] Board list on landing page — `useTable(tables.board)` filtered by owner identity
- [ ] Create board wizard at `/board/new` — calls `create_board` reducer
- [ ] Board detail page at `/board/:id` — client-rendered; subscribes to board + media items
- [ ] Board settings at `/board/:id/settings` — `update_board`, `delete_board`, `regenerate_invite`

### M5 — TMDB search + import (days 12–15)

- [ ] `/api/tmdb/search` Route Handler — proxies TMDB `/search/multi`; API key server-side
- [ ] `AddMediaSearch` component — debounced input, result list, import button
- [ ] Import button calls `import_tmdb_title` procedure; `MediaItem` rows appear in subscription
- [ ] Handles both `MOVIE` and `TV` types
- [ ] TMDB attribution in footer

### M6 — Canvas rendering (days 15–20)

- [ ] `BoardCanvas` component: react-konva `Stage` + `Layer`; fullscreen
- [ ] `lib/canvas-layout.ts`: compute `canvas_x/y` from `chrono_order` and `lane_index`
- [ ] Custom node components rendered from subscribed `MediaItem` rows
- [ ] Edges drawn between chronologically adjacent nodes within a lane
- [ ] Pan (drag stage) + pinch-to-zoom (touchmove two-finger) + scroll-wheel zoom
- [ ] Ordering mode toggle: re-sorts and re-lays out canvas on change

### M7 — Zoom-level collapse (days 20–24)

- [ ] `ZoomController`: tracks current zoom scale; derives active resolution level (1–5)
- [ ] At level < 4: episode nodes hidden, replaced by season node with aggregate summary
- [ ] At level < 3: season nodes hidden, replaced by show node
- [ ] Collapsed node displays: poster + "N episodes • M/P watched"
- [ ] Smooth Konva `Tween` animation on collapse/expand

### M8 — Participant join flow (days 24–26)

- [ ] `/board/:id/join?invite=<token>` page — display name input form
- [ ] On submit: connect without auth token (anonymous identity); call `join_board(board_id, invite_token, display_name)`
- [ ] Anonymous token stored in `localStorage` as `ihw_participant_token_<board_id>`
- [ ] On return visits: token restored, same identity reconnects
- [ ] Participant listed in sidebar once joined

### M9 — Watch tracking (days 26–29)

- [ ] `useWatchState` hook: `useTable(watchEntry)` + `useTable(watchAggregate)` for board
- [ ] Click/tap on a node: toggle watched for current identity via `set_watch` reducer
- [ ] Episode drill-down panel: expand a season/show node to see per-episode toggles
- [ ] Watch propagation: server-side `set_watch` reducer updates `WatchAggregate` for all ancestors
- [ ] Aggregate state used to display partial/full watched status on collapsed nodes

### M10 — Watch overlay + sharing (days 29–32)

- [ ] `WatchIndicator`: row of coloured avatar chips per participant on each node
- [ ] Colour assignment: deterministic from participant `Identity` hash
- [ ] Collapsed nodes show "N/M watched" aggregate count
- [ ] Participant list sidebar (owner view shows all; participant view shows self)
- [ ] Board sharing mode toggle in settings
- [ ] Public route `/board/:id?view=public`: read-only, no watch UI

## 5. Near-term Pre-coding Tasks

Before writing production code:

1. **Confirm SpacetimeAuth Google OAuth setup** — create a Maincloud project; configure SpacetimeAuth with Google provider; verify OIDC token flow in a minimal Next.js prototype before committing to it in the full app.
2. **Procedure beta assessment** — build a minimal `import_tmdb_title` prototype locally to verify that TypeScript procedures can make HTTP calls reliably. If unstable, implement fallback: Next.js Route Handler calls TMDB, then calls the SpacetimeDB HTTP REST API to trigger a `receive_tmdb_data` reducer.
3. **react-konva pinch-zoom** — implement a minimal pinch-to-zoom prototype (`Konva.hitOnDragEnabled = true` + custom `touchmove` handler) before building the full canvas. This is the most mobile-critical interaction.
4. **TMDB API key** — register at [themoviedb.org](https://www.themoviedb.org/settings/api); store in `.env.local` (server-side only, never exposed to client).
5. **Resolve SPEC Q2** (anonymous identity token expiry) before implementing M8.

## 6. Definition of Done — Phase 1

Phase 1 is complete when all of the following hold:

- [ ] A board owner can sign in with Google (via SpacetimeAuth), create a board, and import a TV franchise from TMDB
- [ ] Imported media is displayed as a react-konva node canvas with correct chronological ordering
- [ ] Zoom-level collapse works across at least 3 levels (episode → season → show)
- [ ] A named participant can join via invite link (entering only a display name) and mark titles/episodes watched without creating an account
- [ ] Watch state updates appear in real-time on all connected clients without page refresh
- [ ] Watch state is reflected on canvas nodes with per-participant avatar chips
- [ ] A board can be made public (view-only) or kept private
- [ ] The Next.js app is deployed on Vercel; the SpacetimeDB module is live on Maincloud
- [ ] TMDB attribution is displayed in the footer
- [ ] No secrets (TMDB API key, SpacetimeAuth credentials) are committed to the repository
- [ ] The app is usable on mobile (pinch-to-zoom, tap to mark watched)

- Export board as shareable image
- Board templates (pre-populated franchises)
- Consider Railway self-host migration if Maincloud free tier is insufficient

### Deployment migration path (Maincloud → Railway)

When self-hosting becomes necessary:
1. Deploy SpacetimeDB Docker image on Railway with a persistent volume
2. Re-publish the module to the Railway instance: `spacetime publish --server <railway-url>`
3. Update `NEXT_PUBLIC_SPACETIMEDB_HOST` in Vercel env vars
4. Investigate SpacetimeDB data export/backup strategy (Q7 in SPEC) before migration
