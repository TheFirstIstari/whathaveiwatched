# WhatHaveIWatched — Agent Notes

Sharable/collaborative lists of shows and movies per fandom/IP, designed for watchparties and movie marathons. Lists are displayed in a node-based view showing what individual members of a collaborative workspace have watched. Nodes/shows can be ordered by release date and fandom chronology.

## Stack

- **Backend:** SpacetimeDB Rust module (tables + reducers, self-hosted)
- **Frontend:** Next.js 16 (App Router, Turbopack)
- **Real-time:** SpacetimeDB WebSocket subscriptions
- **Media data:** TMDB API (server-side via `/api/tmdb/*` routes)

## Key architecture notes

- SpacetimeDB reducers are **synchronous** — no HTTP calls inside them. TMDB data is fetched client-side via the Next.js API route `/api/tmdb/fetch`, then passed into `insert_movie` / `insert_tv_show` reducers.
- All reducer calls use **object params** (not positional args), matching the generated TypeScript bindings schema.
- Table accessor names are **snake_case** (`tables.media_item`, `tables.watch_entry`, `tables.watch_aggregate`).
- Auth is anonymous SpacetimeDB identity tokens stored in `localStorage`. No OAuth.
- `frontend/src/module_bindings/` is auto-generated by `spacetime generate` — do not edit by hand.
- `tsconfig.json` target is `ES2020` (required for BigInt literals used throughout).
- Pages using SpacetimeDB hooks are all `'use client'` — `export const dynamic = 'force-dynamic'` does NOT work inside client components (Next.js ignores it). All pages are fully client-rendered.
- `app/providers.tsx` withholds SpacetimeProvider until client mounts to avoid `useSpacetimeDB must be used within a SpacetimeDBProvider` crashes. Pages guard with their own `mounted` state.
- `MediaItem` table does NOT have `episodeNumber`/`seasonNumber` fields. Episode labels in the drill-down drawer are parsed from the title string (e.g. "S01E03 - Title").
- `BoardCanvas` uses a `containerRef` + `ResizeObserver` to size the Konva Stage to its parent `<main>` element, NOT `window.innerWidth/Height`. This prevents the canvas from extending behind the header/footer.
- `DrillDownDrawer` is `z-[60]` (above the board header's `z-10`) and is `position: fixed` so it overlays the full viewport when open.

## Styling — Tailwind v4 (CRITICAL)

- Uses **Tailwind v4** (`tailwindcss@4`, `@tailwindcss/postcss`). `app/globals.css` MUST start with `@import "tailwindcss";` — NOT the v3 `@tailwind base/components/utilities` directives. v3 directives silently produce ZERO utility classes (build still "passes"), which breaks all spacing/layout/responsive while custom `.ui-*` classes + CSS vars keep working. This masquerades as "UI not built correctly."
- Class-based dark mode requires `@custom-variant dark (&:where(.dark, .dark *));` in globals.css (the `html.dark` toggle won't drive `dark:` utilities otherwise).
- v4 auto-detects content (app/, components/); `tailwind.config.ts` is vestigial (not auto-loaded without `@config`).
- v4 utilities compile to `calc(var(--spacing) * N)`, not literal rem — grep accordingly when verifying.
- Visual QA: `node scripts/shoot.mjs` (Playwright/chromium) screenshots all routes × light/dark × desktop/mobile to `/tmp/ihw-shots`.

## Testing

Full pyramid lives in `frontend/`:

- **Unit + component (Vitest, jsdom)** — `*.test.ts(x)` colocated with source.
  - `lib/`: `episodeLabel`, `avatarColor`, `canvas/layout`, `db/connection` (+ legacy compat shims), `db/importMedia` (mocked fetch), `tmdb` (posterUrl), `theme` (token completeness/parity).
  - `lib/hooks/`: `useCanvasCamera` (zoom/fit math), `useKonvaImage` (state machine).
  - `components/ui/`: `Button`, `Input`, `Modal` (Radix Dialog), `Spinner`.
  - Config: `vitest.config.ts`, setup `test/setup.ts` (jest-dom + cleanup + localStorage clear).
  - Run: `npm test` (CI), `npm run test:watch`.
  - **14 test files, 88 tests** — all passing as of June 2026.
  - **Rust module** (`module/src/`): No unit tests — reducers/helpers are tightly coupled to SpacetimeDB's `ReducerContext` and require the framework's integration test harness.
- **E2E (Playwright)** — `frontend/e2e/*.spec.ts`, projects `desktop` + `mobile`.
  - `auth.spec.ts`, `navigation.spec.ts`: signin, guards, dashboard, create-form validation, theme toggle persistence, 404. No DB writes.
  - `board-lifecycle.spec.ts`: real-SpacetimeDB create→verify→delete. **Opt-in** via `E2E_WRITE=1`; skipped otherwise; desktop-only; cleans up after itself.
  - Config: `playwright.config.ts`. Default `webServer` runs `npm run build && next start -p 3100`.
  - `reuseExistingServer` is OFF by default (set `E2E_REUSE=1` to reuse) — a stale `next start` on the port will otherwise serve an OLD bundle and produce bogus failures. Or point `E2E_BASE_URL=http://localhost:3000` at the dev server to skip the build.
  - Run: `npm run test:e2e`, `npm run test:e2e:ui`. All-in-one: `npm run test:all`.
- **Visual QA**: `npm run shots` → `scripts/shoot.mjs` screenshots every route × light/dark × desktop/mobile to `/tmp/ihw-shots`.
- **CI**: `.github/workflows/test.yml` — `unit` job (Vitest) + `e2e` job (Playwright, builds prod, write test NOT enabled). STDB/TMDB build env come from repo secrets with harmless fallbacks.

### Known issues surfaced by tests
- **Real-DB write path fails in headless Chromium**: `createBoard`/`registerOwner` never confirm — browser throws `TypeError: undefined is not a function (near '...chunk of decompressedStream...')` from the SpacetimeDB SDK's decompression. The `E2E_WRITE` lifecycle test correctly fails until this is fixed; investigate SDK message decompression / `DecompressionStream` support.

## Deployment & Render efficiency

- Deployed on Render as a Node web service at `watched.tweak.wiki` (see root `render.yaml`).
- **Health check / keep-warm:** `GET /api/health` (no I/O, no-store). Wired as Render `healthCheckPath`; also a cheap cron-ping target to mitigate free-tier cold starts.
- **Caching:** TMDB proxy routes (`/api/tmdb/search`, `/api/tmdb/fetch`) set `Cache-Control` (CDN `s-maxage` + `stale-while-revalidate`) so repeat queries don't re-hit the origin or TMDB. Do **not** reintroduce a blanket `no-store` in `next.config.ts` — board pages are client-rendered with no per-user SSR data.
- Posters load directly from `image.tmdb.org` via `<img>` (no `next/image`), so there is **no** Render image-optimization cost. Keep it that way.

## Theme

Cinematic terminal palette — warm stone neutrals + teal accent, JetBrains Mono, **light is the default**. CSS-var tokens live in `app/globals.css` (`:root` light, `.dark` override); the Konva canvas mirrors them in `lib/theme.ts` (`LIGHT_THEME`/`DARK_THEME`). Watch-state chips stay color-coded (green/amber) for usability. The two theme files must be kept in sync manually.
