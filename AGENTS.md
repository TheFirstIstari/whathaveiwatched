# WhatHaveIWatched — Agent Notes

Sharable/collaborative lists of shows and movies per fandom/IP, designed for watchparties and movie marathons. Lists are displayed in a node-based view showing what individual members of a collaborative workspace have watched. Nodes/shows can be ordered by release date and fandom chronology.

## Stack

- **Backend:** SpacetimeDB Rust module (tables + reducers, self-hosted)
- **Frontend:** Next.js 16 (App Router, Turbopack)
- **Real-time:** SpacetimeDB WebSocket subscriptions
- **Media data:** TMDB API (server-side via `/api/tmdb/*` routes)

## Key architecture notes

- SpacetimeDB reducers are **synchronous** — no HTTP calls inside them. TMDB data is fetched client-side via the Next.js API route `/api/tmdb/fetch`, then passed into `insert_movie` / `insert_tv_show` reducers.
- All reducer calls use **object params** (not positional args), matching the generated TypeScript bindings schema.
- Table accessor names are **snake_case** (`tables.media_item`, `tables.watch_entry`, `tables.watch_aggregate`).
- Auth is anonymous SpacetimeDB identity tokens stored in `localStorage`. No OAuth.
- `frontend/src/module_bindings/` is auto-generated by `spacetime generate` — do not edit by hand.
- `tsconfig.json` target is `ES2020` (required for BigInt literals used throughout).
- Pages using SpacetimeDB hooks must have `export const dynamic = 'force-dynamic'` to prevent Next.js prerender errors.
- `app/providers.tsx` withholds children from SSR (returns a shell until client mounts) to avoid `useSpacetimeDB must be used within a SpacetimeDBProvider` crashes.

## Environment variables needed

```
NEXT_PUBLIC_SPACETIMEDB_HOST=wss://<host>
NEXT_PUBLIC_SPACETIMEDB_DB=whathaveiwatched
TMDB_READ_ACCESS_TOKEN=<token>
```

## Deployment & Render efficiency

- Deployed on Render as a Node web service at `watched.tweak.wiki` (see root `render.yaml`).
- **Health check / keep-warm:** `GET /api/health` (no I/O, no-store). Wired as Render `healthCheckPath`; also a cheap cron-ping target to mitigate free-tier cold starts.
- **Caching:** TMDB proxy routes (`/api/tmdb/search`, `/api/tmdb/fetch`) set `Cache-Control` (CDN `s-maxage` + `stale-while-revalidate`) so repeat queries don't re-hit the origin or TMDB. Do **not** reintroduce a blanket `no-store` in `next.config.ts` — board pages are client-rendered with no per-user SSR data.
- Posters load directly from `image.tmdb.org` via `<img>` (no `next/image`), so there is **no** Render image-optimization cost. Keep it that way.

## Theme

Monolithic terminal palette (single teal accent + neutral grays), JetBrains Mono, **light is the default**. CSS-var tokens live in `app/globals.css` (`:root` light, `.dark` override); the Konva canvas mirrors them in `lib/theme.ts` (`LIGHT_THEME`/`DARK_THEME`). Watch-state chips stay color-coded (green/amber) for usability.
